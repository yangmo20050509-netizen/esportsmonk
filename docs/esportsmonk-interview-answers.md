# 电竞高僧二面口语答法

下面这份只干一件事：把《电竞高僧项目全经历与二面深挖题库》里的问题，改写成腾讯二面能直接说出口的版本。语气按口语答法写，重点放在判断、取舍、方法和结果。

## A. 产品定义

### 1. 你为什么做这个项目？

我做这个项目，起点很直接。我自己在看 LOL 比赛时，会反复切赛程页、社区、战队资料、选手资料和资讯页，信息很散，打开成本很高。国内有很多讨论场，也有资讯场，但缺一个像 FootMob、BoxBox 那样，把高频观赛信息压成一个固定入口的工具。我想验证的不是“电竞社区还能不能做”，我想验证的是“电竞观赛工具是不是还有明显空位”。

### 2. 这个项目解决的核心痛点是什么？

核心痛点是观赛前的信息割裂。用户真正需要的东西其实很集中，就是今天打什么、比分到哪、主队和关键选手是什么状态、这场值不值得看、赛前大概怎么判断。现在这些信息分散在不同页面和不同平台里，所以我做的是压缩和重组，把高频观赛前要查的一切压到一个地方。

### 3. 为什么定位成观赛工具，不做社区？

因为社区不是一个轻产品问题，是一个重运营问题。社区要解决内容供给、关系链、互动氛围、冷启动和治理，这些都不是现在这个阶段最应该解的题。这个方向如果一开始就做社区，产品很容易散。观赛工具更适合先切，因为它的打开理由更明确，频次更高，用户价值也更直接。

### 4. 为什么只做 LOL，不做多项目电竞？

因为 LOL 最适合做高频观赛工具。赛程密度高、赛事体系成熟、战队和选手关注强、数据结构相对清晰，用户会反复打开。多项目同时做，看起来更大，实际会把数据和产品都做散。这个阶段把一个项目做深，比做一个表面上的“大而全”更值钱。

### 5. 你为什么认为这个方向有机会？

我的判断是，国内电竞内容很多，但稳定的高频观赛入口不够成熟。现在用户要么在社区看讨论，要么在直播平台消费内容，要么在官网零散查信息。中间缺一个“打开就知道今天该看什么”的工具层。这个位置在体育里已经被验证过了，电竞这边还没被做顺，所以我觉得有机会。

### 6. 你对标的是 FootMob、BoxBox 还是别的产品？

我对标的是两类产品。信息结构和高频入口层面，更接近 FootMob。移动端观赛助手和 Widget 心智层面，更接近 BoxBox。它们共同启发我的点，不是谁的 UI 更好看，而是它们都把高频使用场景压到了一个稳定入口。我不是复刻它们的功能清单，我是借它们的产品逻辑来切电竞场景。

### 7. 这个产品最早版本和现在版本的定位有什么变化？

最早版本更像“电竞版 FootMob”的大概念，想同时包含赛程、战队、选手、预测、小组件这些东西。做着做着很快发现，边界太宽了。现在定位已经收窄成一个围绕 LOL 观赛场景的高频入口产品，核心是赛程、比分、主队动态、选手资料和可解释预测。这个变化本质上是从“愿景”收到了“主场景”。

## B. 产品设计

### 8. 首页为什么是现在这套结构？

首页结构是按打开顺序排的，不是按信息种类排的。用户打开后第一眼最关心的是自己关注的主队有没有比赛、状态怎么样、接下来打谁。第二层才是当前比赛和下一场。第三层才是预测和扩展资料。所以首页不是做成导航页，而是做成观赛决策页。

### 9. 为什么主队动态放在最顶上？

因为它是最高频的入口。大部分用户不是随机来查整个联赛，他们先看的是“我主队今天有没有东西”。把主队动态放在最顶上，等于把首页直接变成个性化入口。默认主队是 BLG，本质上是在验证这个模块是不是能成为产品的第一抓手。

### 10. 为什么要把战队页和选手页分开，而不是都堆在首页？

因为这两类信息密度和使用意图不一样。战队页更偏整体趋势和赛季画像，选手页更偏个人风格、英雄池和最近表现。如果都堆在首页，首页会立刻变成信息垃圾场。首页负责“要不要看”，战队页和选手页负责“为什么会这样”，职责要分开。

### 11. 高僧预测为什么只保留一张焦点卡？

因为预测信息天然是强注意力模块，放多了会抢整个站的节奏。首页只留一张焦点卡，是为了让它成为“今天最值得看的一局”的补充判断，而不是把首页做成玄学市场。真正的深度判断放到预测页里看，首页只放一个最能带动点击的入口。

### 12. 你为什么不做用户评论和预测社区？

因为我不想让产品一开始就陷入低质互动。评论和预测社区一旦打开，内容质量、审核、运营、用户关系链都会成为主问题，工具价值反而会被冲淡。这个阶段我更在意的是用户会不会每天打开看赛程和提醒，而不是会不会在里面吵架。

### 13. 你怎么判断哪些信息应该上首页，哪些不该？

我的标准很简单：看它是不是“赛前一眼必须知道”的信息。像主队动态、当前比赛、下一场、焦点预测，这些应该上首页。像详细英雄数据、长篇战队 dossier、历史复盘，这些不该上首页，因为它们不是打开第一屏就必须消费的内容。首页一定要为频次服务，不为信息完整性服务。

## C. 数据与工程

### 14. 这个项目的数据到底从哪里来？

现在主要是三层数据源。第一层是腾讯英雄联盟赛事公开赛程文件，负责赛程、比分、赛事阶段这些核心官源信息。第二层是腾讯选手详情页，负责定妆照、常用英雄、选手基础数据。第三层是我自己整理的知识层，包括视频 OCR 结构化知识库、新闻分析库和战队选手 dossier。这三层职责是分开的。

### 15. 你为什么没有直接上 GRID / Riot 官方电竞整套数据？

因为理想主源和可落地主源不是一回事。GRID 和 Riot 官方电竞数据当然更完整，但这个项目是个人产品，不是已经签好权限的商业合作项目。我要的是先把能跑通、能验证、能稳定更新的版本做出来，而不是一开始卡死在授权和接入门槛上。所以我先用可确认官源把 MVP 跑起来，再为更强主源留接口空间。

### 16. 腾讯赛事页和腾讯选手页分别承担什么角色？

腾讯赛事页承担赛程和比分主源的角色，核心目标是“今天打什么、打到哪”。腾讯选手页承担资料和画像主源的角色，核心目标是“这个人是谁、常玩什么、最近状态怎样”。一个是赛事层，一个是人层，它们解决的是两类不同的信息问题。

### 17. 你怎么保证赛程和比分的可靠性？

我做了两件事。第一，核心赛程和比分只认官源，不拿社区和第三方站的二手信息做主判断。第二，构建时把数据压成静态产物，站点展示层不直接拼接乱七八糟的数据请求。这样能把线上不稳定因素压下去，至少不会出现用户打开首页发现赛程和比分乱跳的情况。

### 18. 如果某个数据源挂了，产品会怎么退化？

我现在的思路是分级退化。最关键的赛程和比分如果挂了，站点至少还能展示上一轮构建成功的静态数据，不会整站白屏。选手资料挂了，最多是选手页内容变薄。知识库或者 Gemini 挂了，高僧预测会退回规则层和 fallback 文案。也就是说，退化是局部的，不是整个站直接死。

### 19. 为什么你一开始没有做常驻后端？

因为项目初期最值钱的是验证产品，不是背服务器债。这个站本质上是静态站加构建期数据整合，用 Git 直连和构建脚本就能跑通大部分链路。常驻后端要带来部署、监控、缓存、成本和额外复杂度，这些在早期是负担，不是增益。

### 20. 你现在这套 Git + EdgeOne 的构建模式，解决了什么问题？

它解决的是“手工上传静态站”的不稳定和不可追溯。现在代码 push 到 GitHub，EdgeOne 自动拉取、构建、发布，谁改了什么、哪次上线用了哪个 commit，全部可追。它让这个项目从“手工页面作品”变成了一个最小可持续发布系统。

### 21. `/api/site-data` 和静态 `site-data.json` 双通路为什么要共存？

因为两条线解决的是两个问题。实时 API 解决“打开页面就尽量拿到最新数据”。静态 JSON 解决“就算实时口挂了，页面也还能看”。两条共存的目的不是炫技，是为了让产品既能新，又能稳。纯实时容易炸，纯静态又会老，所以我用了双通路。

### 22. 你踩过的最恶心的部署坑是什么？

最恶心的一次是 EdgeOne 函数目录识别问题。站点内容已经能更新，但 `/api/site-data` 一直 404，看起来像代码问题，实际是平台函数目录约定和构建产物目录没对上。我最后是把发布产物、本地运行、平台目录规则全拆开看，才把这条链抠通。这个坑很烦，因为表面上是 404，根因其实在部署约定。

### 23. 为什么最终选择 Git 直连自动构建，而不是手动上传？

因为手动上传只适合一次性演示，不适合持续迭代。手动上传会反复出现目录传错、线上版本落后、本地和线上不一致这些低级事故。Git 直连以后，代码、构建、发布都变成一条可重复链路，这才有产品味。

## D. AI 与知识库

### 24. 高僧预测到底是怎么工作的？

现在是三层。底层是规则层，算账面、近势、局差、交手、休整、BO、英雄池这些东西。中层是知识层，补战队门风、选手手法、最近分析和视频知识库。最上层才是大模型，把结构化结果写成高僧见解。这个顺序很关键，不然模型会开始胡扯。

### 25. 模型在里面负责什么，不负责什么？

模型负责表达，不负责最终判断。它负责把规则层和知识层的结果写成用户愿意看的断语，负责形成风格和阅读体验。它不负责直接拍脑袋算胜负，也不负责改写赛程、比分和基础事实。事实层和数值层我不交给模型。

### 26. 为什么你没有让大模型直接预测胜负？

因为直接让模型猜比赛，短期看起来有戏，长期一定翻车。比赛预测需要稳定特征、赛前快照和赛后回填，而不是单次语言推理。我更信规则层给出可解释的判断，再让模型去写，而不是让模型一边算一边编。

### 27. 你怎么避免模型写出“装懂”的废话？

我做了三件事。第一，提示词里明确写它必须讲长板、毛病和翻盘路径。第二，给它喂结构化知识，不让它全靠空想。第三，做了黑名单和 fallback，把那些翻译腔、假文艺和空词都压掉。所以现在它写出来的东西，至少会落到比赛逻辑上，而不是泛泛抒情。

### 28. 蓝宝石视频知识库是怎么进来的？

我是先把视频转成文字，再把 OCR 脏文本做清洗，纠正战队、选手、英雄名，最后抽成结构化条目。真正值钱的不是原文，而是从原文里抽出来的门风、强项、毛病和翻盘条件。这样它才能进入预测链，而不是变成一堆只适合阅读的散文。

### 29. OCR 脏数据怎么清洗成结构化知识？

清洗分三步。先做实体纠错，把错识别的人名、队名、英雄名拉正。再做语义拆分，把一段话拆成队伍风格、选手特点、赢法、发病点。最后把这些信息落成可被程序消费的字段，不是留在自由文本里。这样后面模型和规则层都能真用。

### 30. 英雄池和选手资料现在是怎么影响预测的？

它们现在已经不只是展示层信息了。英雄池深度、舒适英雄覆盖、近期切牌会进分数核；选手资料会进入知识层，让模型知道关键位该看什么刀口。所以它们现在既影响“怎么算”，也影响“怎么说”。

### 31. 你说项目会“越来越懂比赛”，这个增长机制具体是什么？

我的思路是做赛前快照和赛后回填。每场比赛开打前，把当时可见的赛程、战绩、英雄池、知识因子和高僧预测存下来。比赛打完，再把真实结果和命中情况回填进去。积累多了以后，知识库能变厚，因子权重能校准，模型也能看到更长的上下文，这才叫越来越懂。

### 32. 你考虑过 fine-tuning 吗？为什么现在还没做？

考虑过，但现在不急着做。因为当前最缺的不是模型参数，而是高质量训练样本。现在更值钱的是把赛前快照、赛后回填、知识库和 dossier 做厚。等样本量够了，再考虑把文风和偏好做成 fine-tuning。这个顺序更稳，也更省成本。

## E. 项目管理与取舍

### 33. 你最困难的一次产品取舍是什么？

最困难的一次取舍是把“电竞版 FootMob”的大概念收窄成观赛工具。因为一开始什么都想做，看起来很爽，但做下去会发现每条线都很重。最后我把社区、多项目电竞和复杂互动都砍掉了，留下赛程、比分、战队、选手和预测。这个取舍痛，但对。

### 34. 你砍掉过哪些功能？为什么砍？

砍掉过用户评论、预测社区、多项目电竞、复杂实时局内数据、过重的首页信息堆叠。这些功能不是永远不做，是当前阶段不该做。它们会分散产品焦点，还会把工程复杂度和运营复杂度一起拉起来。

### 35. 这个项目里最费时间的不是写页面，那是什么？

是把数据、知识和预测链真正拧成一套系统。页面反而是最直观的，最费时间的是数据源筛选、构建链、自动更新、知识库整理、模型文案治理这些看不见的基础层。这些东西不做好，页面再漂亮也只是壳。

### 36. 如果只给你两周，你会砍成什么样？

我会只保留三件事。第一，主队动态和赛程比分。第二，战队页和选手页的最小资料。第三，一张焦点高僧预测卡。知识库、小程序、更多队伍、复杂因子都先压后。两周内最该保证的是可用和稳定，不是完整。

### 37. 如果继续往前做三个月，你会优先补哪三件事？

第一，赛前快照和赛后回填，把预测闭环做实。第二，知识库继续扩到更完整的战队和选手 dossier。第三，原生 iOS 端和 Widget，把高频入口做出来。这三件事分别对应“更准”“更懂”“更常被打开”。

### 38. 你怎么验证这个项目是不是有人愿意长期使用？

我不会先看注册量，我会先看复访。重点看三个指标：一是用户会不会在比赛日前反复打开。二是订阅提醒有没有被真正点开。三是主队切换和选手页的点击深度。如果这些行为起不来，再漂亮的页面都没意义。

### 39. 收费你会怎么做？

当前阶段我更倾向免费验证主场景。等用户真的形成使用习惯，再考虑卖增强能力，比如更早的预测、更细的因子拆解、主队专属订阅、深度复盘。基础赛程和比分不该收费，收费应该落在“更高频、更深、更早”的能力上。

### 40. 小程序和 iOS 原生端你会怎么分工？

小程序更适合分发、社群传播和订阅消息，适合做用户承接。iOS 原生端更适合做高频打开和 Widget，适合做核心体验。我的分工思路是，小程序先承担增长和消息触达，原生端承担高频观赛工具心智。

## 一句收口

这项目如果在二面里要我自己概括，我会说：我不是单纯做了一个电竞站，我是把一个跨产品、数据、AI 和前端发布的问题，从概念推到了能跑、能讲、还能继续长的阶段。
