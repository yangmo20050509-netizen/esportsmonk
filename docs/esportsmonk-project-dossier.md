# 电竞高僧项目全经历与二面深挖题库

## 一、项目一句话

电竞高僧是一个围绕英雄联盟观赛场景搭建的个人产品，目标是把原本分散在赛事官网、社区、资讯、战队页和选手页里的信息压缩成一个高频打开、低干扰、带判断能力的观赛入口。产品当前覆盖官网站点、自动更新的数据构建链路、赛前“高僧预测”、选手与战队档案、小程序壳子，以及一套持续生长的知识库。

它不是社区，也不是大而全的数据站。它切的是“高频观赛辅助”这个更窄但更具体的入口。

## 二、项目缘起

这个项目最早来自一个很明确的判断：

1. 国内有虎扑、微博、直播平台这类讨论和消费内容的场子。
2. 但缺一个像 FootMob、BoxBox 那样，能把赛程、比分、战队、选手、提醒和赛前判断压成一个稳定入口的电竞工具。
3. 英雄联盟又是电竞里最适合做这类产品的板块，因为赛程密度高、用户打开频率高、战队和选手的信息消费很强。

最早的目标也很收敛：

- 不做社区
- 不做所有电竞项目
- 不做复杂 UGC
- 先只做 LOL
- 先把“打开就能看懂今天该看什么、为什么值得看、哪边更值得押”这件事做顺

这一步很关键。因为如果一开始就想做社区或者做全品类电竞，产品会直接散掉。

## 三、项目核心定位

项目经历里最该讲清的不是“我做了一个网站”，而是“我把产品切在哪”。

电竞高僧的定位经历过几次收敛：

### 1. 最初概念期

最早的想法比较宽：

- 赛事看板
- 战队页
- 选手页
- 比赛预测
- 小组件 / App

这时候更接近“电竞版 FootMob”的愿景。

### 2. 第一次收敛

在真正推进时，很快发现社区和内容讨论不该现在碰，原因很现实：

- 社区是重运营产品
- 冷启动极难
- 内容供给和关系链要求高
- 会拖死数据和工具链建设

所以产品被收缩成：

- 高速查看赛程
- 快速知道主队和主选手状态
- 在赛前给出一段可分享、可讨论的判断

### 3. 当前定位

当前最稳定的定义是：

> 一个围绕 LOL 观赛场景的高频入口产品，用官方赛程和可解释预测把“看比赛前要查的一切”压缩到一个地方。

## 四、从概念到落地的完整过程

### 阶段 1：先把方向判清

这一阶段主要解决两个问题：

1. 产品究竟做什么
2. 数据究竟从哪来

一开始最大的不确定性不是前端，而是数据。

因为电竞产品要是真的能用，赛程和比分不能错。只要错一场，用户就会默认你整个产品不可靠。

所以第一阶段做的事情是：

- 明确产品边界
- 梳理数据源
- 评估官方链路和野路子链路
- 判断什么能做 MVP，什么做不了

这个阶段产出了几类关键判断：

#### 产品判断

- 先不做社区
- 只做英雄联盟
- 优先做高频观赛工具
- 预测模块要做，但一定要可解释

#### 数据判断

- 职业赛赛程和比分必须优先走官方或准官方源
- Riot / GRID 是理想主源，但拿权限不现实
- 公开网页和腾讯赛事页能支撑 MVP
- Rank、选手页、英雄池这些可以从 Riot 或腾讯选手页侧面补

这一步其实决定了项目后面能不能继续。

### 阶段 2：先做出能看的静态原型

第二阶段先做了一个高保真静态站雏形，用来验证信息结构和页面骨架：

- 首页
- 战队页
- 选手页
- 高僧预测页

这个阶段的重点不是代码质量，而是把核心信息结构定死：

- 首页该先看什么
- 战队页该放什么
- 选手页到底要不要 Rank
- 预测是长卡还是短卡
- 用户到底会不会在一屏内看完

这时候做过很多前端方向试错：

- 氛围感强的暖光背景
- 流动感背景
- 更像 BoxBox 的信息卡
- 更像产品站的白底纸面结构

这个阶段最后得到的结论是：

- 背景可以有气氛，但不能喧宾夺主
- 信息密度必须克制
- 不能做成 AI 生成味很重的花里胡哨首页
- 观赛工具更适合接近“编辑页 + 看板”的混合结构

### 阶段 3：做出自己能用的手机版 MVP

第三阶段把目标压到了“先给自己用”。

这一步的思路很实际：

- 先别幻想大规模产品
- 先做一个自己会天天打开的观赛助手
- 默认主队 BLG，默认选手 Bin
- 在中国网络环境可用

所以这时候做了几件事：

1. 自用手机页
2. Scriptable 小组件
3. H5 深链
4. 只做腾讯官方赛程快照

这一步很重要，因为它验证了一个事实：

> 真正有价值的不是“我有一个完整产品概念”，而是“我自己会不会每天打开它”。

这个阶段也暴露了一个现实：

- iOS 桌面小组件很强
- 但公开卖就得进 App Store 和 WidgetKit 体系
- 成本和门槛都更高

所以这条线后来被拆成：

- 先官网 / 小程序 / 自用工具
- 再考虑原生 iOS

### 阶段 4：官网从静态 demo 变成真数据站

这是项目第一次真正复杂起来的阶段。

因为从“写死数据 demo”变成“真数据官网”，本质上是产品进入了工程状态。

这一阶段陆续做掉了：

1. 官方赛事数据接入
2. 真数据构建脚本
3. 发布产物
4. 线上自动更新链路
5. 站点风格重构

核心成果：

- 官网不再只读 mock
- 站点数据统一从构建产物 `site-data.json` 来
- 后来又增加了同域 API 与静态兜底双通路
- EdgeOne 接入 Git 直连，代码 push 后站点自动更新

这一步意味着项目已经不只是“页面”，而是一个可持续更新的产品壳。

### 阶段 5：高僧预测从 gimmick 变成一套结构

“高僧预测”最开始只是概念，后来逐步拆成了三层：

1. 数值层
2. 因子层
3. 文案层

#### 数值层

最开始吃的是：

- 战绩
- 近五场
- 局差
- 交手
- 休整时间
- BO 信息

后来逐步加入：

- 英雄池深度
- 舒适英雄覆盖
- 近期切牌
- 选手资料
- 战队与选手 dossier
- 结构化知识库

#### 因子层

把赛前判断拆成明示的因子：

- 账面
- 近势
- 局口
- 旧账
- 歇脚
- 门风
- 刀口

这样它不是一段玄学，而是有因有据。

#### 文案层

这是最容易翻车的一层。

早期模型写出来的东西很容易：

- 假文艺
- 翻译腔
- 不懂装懂
- 讲废话

所以后来这块做了几轮彻底重写：

- 提示词重写
- 黑名单词汇
- 队伍与选手知识库加厚
- fallback 文案重写
- 高僧口吻与比赛理解分离

最后才得到当前这条路：

> 规则层算判断，知识库给上下文，模型负责把判断写成可读又有风格的断语。

### 阶段 6：知识库和持续生长能力

项目后来最大的升级点，不是页面，而是“开始真正变懂”。

这里做了三层知识输入：

#### 1. 腾讯选手页抓取

拿到的内容包括：

- 定妆照
- 常用英雄
- 历史英雄数据
- KDA
- 伤害
- 分均经济
- 参团率
- MVP

这让选手页和预测不再只靠空文案。

#### 2. 蓝宝石视频 OCR 结构化

把视频转录文字中的脏 OCR 内容清洗后，抽成：

- 队伍门风
- 长板
- 发病点
- 翻盘条件
- 选手手法

这里最大的价值不是“多一份资料”，而是把口水内容变成了结构化知识。

#### 3. 新闻与解读库

又加了一层比赛解读库，用于补：

- 最近舆情
- 赛后复盘
- 当前赛段趋势
- 战队近期状态

然后在构建期把这些资料一起送进高僧预测。

这意味着高僧预测开始具备“持续生长”的基础，而不只是固定模板。

### 阶段 7：小程序与多端承接

在官网稳定后，又起了微信小程序壳：

- 首页
- 战队页
- 选手页
- 高僧预测页
- 订阅消息申请链路

这一步不是为了替代官网，而是为了做：

- 国内更强的分发
- 订阅提醒
- 未来分享卡和消息回流

也说明产品思路已经从“一个站”往“一个可承接多端的工具产品”走了。

到现在为止，电竞高僧已经不是概念了，已经具备这些能力：

### 官网端

- 首页主队动态
- 当前对局 / 下一场
- 赛段榜单
- 战队页
- 选手页
- 高僧预测页

### 数据端

- 腾讯赛事赛程与比分
- 腾讯选手页资料

### AI 端

- 结构化知识喂料

## 六、这个项目里最大、最复杂的问题

### 最大问题：在没有稳定官方后端权限的前提下，把电竞看板做成“看起来像实时产品”的可用站点

这个问题复杂在它同时是产品问题、数据问题和工程问题。

#### 为什么难

1. 你要的是“赛程、比分、选手、预测”一整套，不是一张静态页。
2. 你没有正式拿到 GRID / Riot 电竞完整实时权限。
3. 官网必须稳定，数据错一场就死。
4. 站点还要在中国网络环境下尽量能用。

#### 我怎么拆

我把它拆成了三层：

1. **可确认官源层**
   - 赛程、比分、赛事阶段
2. **二级资料层**
   - 选手资料、定妆照、英雄池
3. **推理层**
   - dossier、知识库、高僧预测

这样每一层的可靠性和职责都不一样，不会互相污染。

#### 真正最难的地方

不是抓数据本身，而是：

> 怎么让一堆半公开、结构不统一、平台行为不稳定的数据源，最后落成一个对用户稳定的产品。

这件事最后靠的是：

- 构建期整合
- 静态产物兜底
- 同域 API 尝试
- Git 驱动自动更新
- 本地知识库补足上下文

### 第二个大问题：怎么让 AI 看起来真懂，而不是像翻译器和神棍的混合体

这个问题比“接个模型”难得多。

因为大模型天然会：

- 写废话
- 装深刻
- 混淆事实和判断
- 用空词糊弄用户

所以我最后把它拆成：

1. 规则层给分
2. 知识库给料
3. 模型只负责把料写出来

这是项目里很值得讲的一点，因为这不是简单调 prompt，而是产品层面的 AI 角色拆分。

## 七、最引以为傲的部分

### 1. 我没把它做成一个“功能拼盘”

这个项目最值得讲的，不是页面数量，而是产品切得够准：

- 不做社区
- 不做全电竞
- 不做内容平台
- 只做高频观赛入口

这让项目始终有焦点。

### 2. 我把数据、内容、AI 三条线拧到了一起

很多个人项目只有一条线：

- 只有页面
- 或只有 AI
- 或只有抓数据

电竞高僧比较完整的地方在于：

- 数据是真的
- 页面是能用的
- AI 不是摆设
- 小程序也已经起了壳

### 3. 我把“越来越懂”这件事做成了路线，而不是一句口号

高僧预测不是固定模板，而是有演进路线：

- 赛前快照
- 赛后回填
- 结构化知识库
- 新闻解读库
- 战队 / 选手 dossier
- 英雄池入算

这说明项目不只是一个 demo，而是已经有了方法论。

## 八、这个项目暴露出的短板

这部分最好也能自己先讲，不然二面问出来会很被动。

### 1. 数据源还不够彻底

当前最核心的赛程比分已经能跑，但职业赛全量官方实时链路还没完全拿下。  
这意味着产品现在更像“强可用的半自动看板”，还不是彻底工业级的数据产品。

### 2. AI 的训练闭环刚起步

现在已经有知识库、赛前快照、历史断局，但还没有做到真正意义上的：

- 自动赛后复盘
- 因子命中率回填
- 参数持续校准
- fine-tuning

### 3. 多端还没完全打通

官网和小程序都在，iOS 原生端还没开始。  
这个会影响“高频打开”的体验上限。

## 九、如果二面深挖，这个项目最值得怎么讲

别按“我做了一个网站”讲。  
更好的讲法是：

> 我用这个项目把一个模糊的想法，做成了一个真实可用的多端观赛工具雏形。里面最有价值的不是页面本身，而是我把产品边界、数据链路、AI 角色拆分和自动更新机制一步步做出来了。

讲这个项目时，建议主线按下面四句走：

1. 我为什么做它
2. 我怎么定义它不是社区而是观赛工具
3. 我怎么解决数据和 AI 这两个最容易翻车的问题
4. 我把它做到哪一步，还差哪一步

### F. 你最该准备的高质量回答方向

这些问题别只背答案，最好按“判断 -> 做法 -> 结果 -> 反思”四段式回答。

#### 1. 讲最大难题时

建议重点讲：

- 数据源不完整
- 官方权限拿不到
- 还得做成稳定可用
- 你最后怎么用构建链、静态兜底和知识库扛住

#### 2. 讲 AI 时

建议重点讲：

- 我没有把 AI 当万能预测器
- 我把数值判断和文案表达拆开了
- 我让知识库和赛后回填成为它“变懂”的基础

#### 3. 讲产品判断时

建议重点讲：

- 我刻意没做社区
- 我故意把产品切窄
- 这样反而更容易形成高频场景

#### 4. 讲工程时

建议重点讲：

- 这不是纯前端站
- 也不是传统后端系统
- 是一套围绕数据构建、自动发布、轻服务化演进的产品基础设施

## 十一、面试时可以主动抛出的亮点

下面这些话，不是让你背，是提醒你主动占住话语权。

1. 这个项目最难的不是页面，而是数据和 AI 的边界怎么拆。
2. 我没有直接让大模型猜比赛，我先把规则层和知识层建起来了。
3. 这个产品已经从静态 demo 变成了可自动更新的产品站点。
4. 我不是只做了官网，还把小程序和后续 iOS widget 路线一起想清了。
5. 这个项目最值钱的不是完成度，而是它已经有“越来越懂”的能力设计。

## 十二、当前阶段最真实的项目总结

电竞高僧现在最准确的定义，不是成熟商业产品，也不是单纯的作品集页面。

它是一个已经完成以下闭环的个人产品：

- 产品定位闭环
- 信息结构闭环
- 数据接入闭环
- 自动构建与发布闭环
- 初步 AI 推理闭环
- 多端延展路线闭环

它还没完成的，是更重的那部分：

- 全自动高质量知识更新
- 真正赛后复盘反哺预测
- 更强的数据主源
- iOS 原生端和 Widget
- 正式商业化闭环

但从面试角度看，这已经不是“我写了个 demo”，而是：

> 我从 0 到 1 把一个复杂度横跨产品、数据、AI、前端、部署和多端分发的项目，真正做到了能跑、能讲、能继续长。
