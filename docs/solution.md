# 电竞高僧 V0 方案

## 产品判断

这个方向能做，但入口别搞歪。

- 你切的不是“电竞社区”，是“高频观赛工具 + 轻内容表达”
- 对标对象该拆成两类：Footmob 的工具效率，虎扑的讨论热度
- 第一阶段别碰社区，社区太吃内容供给和关系链，早期会把产品节奏拖死

## 核心用户

1. 高频看比赛的人，需要一个比微博、虎扑、直播平台更干净的赛程和资料入口
2. 赛前爱聊胜负、版本、对位的人，需要一套能转发的预测卡
3. 国际赛期间会频繁查战队和选手状态的人，需要一眼能看懂的资料页

## MVP 只做什么

1. 首页
   赛程、直播、即将开赛、重点场次提醒
2. 战队页
   近期战绩、阵容、版本强项、交手记录
3. 选手页
   职业赛记录、英雄池、Rank 快照
4. 高僧预测
   概率、证据因子、风险标签、节目化文案

## 暂时别做什么

1. 社区发帖和评论
2. 用户生成预测
3. 复杂社交关系链
4. 全项目制覆盖所有电竞项目

## 数据接入

### 推荐组合

1. 职业赛基础数据
   首选 PandaScore，接入快，覆盖比赛、战队、选手、赛事结构，适合 MVP
2. 更强的实时与官方级数据
   后续看 GRID，延迟和稳定性更强，但商务门槛更高
3. Rank 数据
   用 Riot Games API 的 `league-v4`、`match-v5`，前提是你维护职业选手账号映射

### 具体分工

- PandaScore / GRID
  负责赛事、赛程、战队、选手、系列赛结果、历史比赛
- Riot API
  负责韩服/国服外服账号 Rank、近期对局活跃度、英雄使用分布
- 自建数据库
  负责统一 ID、缓存、快照、预测特征和前端读模型

### 别碰的坑

- 直接爬虎扑、掌盟、LoL Esports 页面，法务和稳定性都恶心
- 直接拿 Rank 数据推职业胜负，相关性有，但绝对不能当主因
- 不做数据快照，后面战绩回溯和预测复盘会一团烂账

## AI 接入

### 正确姿势

AI 只负责两件事：

1. 把结构化特征翻译成用户看得懂的赛前判断
2. 生成有风格但不越界的“高僧预测”文案

### 别让模型碰的地方

- 原始数据清洗
- 概率直接裸算
- 事实判断的唯一来源

### 一条能落地的预测流水线

1. 数据层汇总特征
   近期 10 场、对位、版本、红蓝方、首发、Rank 活跃度、交手记录
2. 规则层做 baseline
   先用简单权重模型给初始概率，模型不背锅
3. LLM 做解释
   输出 `winner`、`confidence`、`factors`、`riskTags`、`monkLine`
4. 前端分层展示
   概率是概率，骚话是骚话，证据因子必须单列

### 为什么这么做

纯 LLM 猜比赛很容易一本正经地放屁。你要的是可复盘的预测产品，不是随机玄学机器。

## 信息架构

1. 首页
   Live, Upcoming, Rankings, Spotlight
2. 战队页
   Overview, Roster, Match History, Meta Fit
3. 选手页
   Overview, Rank Snapshot, Champion Pool, Match Log
4. 预测页
   Match Cards, Confidence, Risk, Monk Lines, Refresh History

## 商业化顺序

1. 先把高频观赛入口做出来
2. 再靠预测卡和赛前前瞻做分享传播
3. 数据稳定后再碰会员、订阅提醒、桌面组件和高级洞察

## 关键指标

1. 首页 DAU / WAU
2. 重点比赛提醒点击率
3. 战队页和选手页二跳率
4. 预测卡分享率
5. 预测卡点击后停留时长
6. 赛前 24 小时到开赛前 10 分钟的回访率

## 6 周 MVP 节奏

1. 第 1 周
   数据源验证，统一实体模型，做赛程和战队基础卡
2. 第 2 到 3 周
   首页、战队页、选手页打通
3. 第 4 周
   预测特征、简单 baseline、结构化 AI 输出
4. 第 5 周
   开赛提醒、分享卡、预测复盘
5. 第 6 周
   打磨视觉和移动端体验，准备小范围测试

## 当前 demo 的定位

这版 demo 的目标很明确：

- 先证明产品长什么样
- 先把信息密度和风格定下来
- 先把 AI 和数据接入的边界画清楚

你要真往下做，下一步就该开始定实体模型和数据表了，别急着先堆页面。
